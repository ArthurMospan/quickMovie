# Захист Firestore (обов'язково зробити один раз)

Зараз будь-хто з анонімною сесією може читати й писати **будь-які** документи
`users/*` (uid виду `tg_<id>` легко перебрати). Код уже готовий до захищеного
режиму — треба лише виконати кроки нижче. Поки кроки не виконані, все працює
як раніше (анонімний fallback), нічого не ламається.

## Крок 1. Службовий ключ Firebase

1. [Firebase Console](https://console.firebase.google.com) → проєкт QuickMovie
   → ⚙️ Project settings → **Service accounts**.
2. **Generate new private key** → завантажиться JSON-файл.

## Крок 2. Env-змінна у Vercel

У проєкті Vercel (той, що з доменом `quick-movie-wheat.vercel.app`):
Settings → Environment Variables → додати

| Name | Value |
|---|---|
| `FIREBASE_SERVICE_ACCOUNT` | увесь вміст JSON-файлу одним рядком (або base64 від нього) |

Потім **Redeploy**.

## Крок 3. Перевірка

1. Відкрий Mini App через бота.
2. Збережи фільм ❤ і переконайся, що він з'явився у вішлісті з іншого
   пристрою / після перезапуску (тобто синк через Firestore працює).
3. (Опційно) `GET https://quick-movie-wheat.vercel.app/api/auth` має віддати
   405 (метод не той) — а не 503. 503 = змінна не підхопилась.

Що відбувається під капотом: апка тепер шле `initData` на `/api/auth`,
сервер перевіряє підпис Telegram (HMAC з токеном бота) і видає Firebase
custom token з uid `tg_<id>`. Анонімний вхід лишається фолбеком.

## Крок 4. Увімкнути правила

Firebase Console → **Firestore Database → Rules** → вставити вміст файлу
`firestore.rules` з кореня репо → **Publish**.

З цього моменту:
- писати у свій документ може лише власник (перевірений через Telegram);
- у чужому документі можна змінити лише `partnerId` і лише «на себе»
  (прийняти запрошення) або «з себе» (відв'язатись);
- видаляти документи не може ніхто.

## Крок 5 (після тижня-двох). Вимкнути анонімний вхід

Коли всі активні юзери хоч раз зайдуть (отримають перевірений вхід):
Firebase Console → Authentication → Sign-in method → **Anonymous → Disable**.

⚠️ Не вимикай анонімний вхід раніше — юзери зі старим кешем втратять синк
до наступного відкриття апки.

## Нотатки

- Крон нагадувань (`/api/cron-reminders`) автоматично перейде на service
  account (він обходить rules). Поки змінної нема — працює старим шляхом,
  але тільки до ввімкнення правил.
- `/api/auth` відхиляє initData старші за 24 години і невалідні підписи.
